## Trace Framing

This card is not independent external validation.

It is an internal tool-architecture trace showing a host-body or exoskeleton lane where continuity, module expansion, and cross-module context problems are handled through layered memory, bounded roles, and explicit handoff surfaces.

## Direct Source Claim

The sources explicitly document:

- ChattyCog as an everyday-use cognitive prosthetic or exoskeleton variant
- GPU main AI plus CPU-only Bookkeeper role split
- a tri-helix memory stack of Hot, Luke Warm, and Cold memory
- cross-module context awareness through department status updates and bridge-mediated handoffs
- module expansion through manifest-based drop-in modules and hosted UI surfaces
- sandbox, ledger, preload, and bridge layers for keeping context and task state tractable

## Source-Grounded Reconstruction

The mechanism trace that can be reconstructed is:

- stuck/problem: a local AI host with modules, longer tasks, and multiple work surfaces risks context loss, memory sprawl, and weak cross-module continuity
- vaulting/deferral: cold logs, Luke Warm rollups, department rundowns, bridge snapshots, scratchpad, and task ledger preserve unresolved context in layered forms rather than forcing full transcript replay
- pivot: instead of a single monolithic chat surface, the architecture shifts toward a host-body model with explicit memory tiers, module handoffs, and bounded helper roles
- catalyst/topology fit: CPU-only Bookkeeper, module bridge, sandbox preload, task ledger, and hosted module surfaces fit the continuity problem better than a flat one-window assistant
- unlock: cross-module context and longer task continuity become more metabolizable without stuffing raw history back into the prompt
- adjacent unlock/cascade: module expansion, nearby-peer workflow sharing, hosted UI lanes, and future memory/context routing surfaces become easier to layer onto the host
