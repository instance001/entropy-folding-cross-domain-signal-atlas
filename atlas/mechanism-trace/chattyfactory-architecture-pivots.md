## Trace Framing

This card is not independent external validation.

It is an internal tool-architecture trace showing how repeated build/patch failures, constraint pressures, and reliability problems appear to drive deferred work, pivots, and later unlocks.

## Direct Source Claim

The sources explicitly document:

- broad all-in-one model work is not reliable enough
- bounded lanes and deterministic host machinery are preferred
- fallback gaps are journaled
- older families are transitional rather than sacred
- starters, host-owned freezing, adaptive decomposition, and automatic decomposition inference became the new direction
- receipts, reviews, x-rays, and governance surfaces preserve failure evidence

## Source-Grounded Reconstruction

The mechanism trace that can be reconstructed is:

- stuck/problem: broad project generation and patch surgery are too unreliable or unsafe if left too unconstrained
- vaulting/deferral: fallback gaps, journals, frozen intent, receipts, project state, and transitional family history preserve unresolved build problems rather than erasing them
- pivot: the system moves from broad positive lanes toward starters, atomized microtasks, adaptive decomposition, and host-owned governance
- catalyst/topology fit: host-owned primitives, decomposition grammars, diagnosis-aware patch surgery, and deterministic contracts fit the unresolved failure patterns better than the earlier broad approach
- unlock: previously too-broad tasks become metabolizable as smaller bounded tasks
- adjacent unlock/cascade: one successful decomposition pattern becomes a direction for broader architecture change and additional governed surfaces

## Mechanism Elements Visible

- stuck / blocked / unresolved tension: clear
- vaulting / latent storage / preserved residue: clear
- pivot / changed context / changed strategy: clear
- catalyst or topology fit: clear
- rapid unlock / metabolization: clear at the task-decomposition level
- adjacent unlock / cascade: reasonably clear
